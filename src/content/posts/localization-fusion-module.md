## 0️⃣ 프로젝트 개요

- **프로젝트명**: 위치 추정 통합 모듈(LFM) 개발
- **프로젝트 기간**: 2025년 1월 - 2025년 4월
- **프로젝트 목적**: TF 생성 모듈과 센서 입력 모듈을 구분하여 모듈 상태 체크 및 위치 추정 안정성 향상
- **팀 구성**: 2명 (ROS 개발자 2명)

## 1️⃣ 역할 및 담당 업무

- **역할**: Localization Fusion Module 개발 총괄
- **담당 업무**:
  - robot_localization 기반 센서 융합 아키텍처 재설계 (AMCL 통합)
  - 레거시 위치추정 시스템 통합 및 마이그레이션
  - 실시간 센서 상태 진단 모듈 설계·구현
  - 실제 AMR 환경에서의 TF 안정성 및 정확도 검증 수행

## 2️⃣ 기술 스택

| 분류 | 기술 | 활용 내용 |
|------|------|-----------|
| **환경** | ROS1 Melodic / Ubuntu 18.04 | 로봇 미들웨어 및 개발 환경 구성 |
| **언어** | C++ | 위치추정 모듈 전체 파이프라인 구현 |
| **핵심 패키지** | robot_localization, AMCL | EKF 기반 센서 융합 및 파티클 필터 위치추정 |

## 3️⃣ Trouble-Shooting

> [!WARNING]
> **[문제 상황 1] Odom 데이터 단절 시 `TF_OLD_DATA` 에러 발생**

- **원인 분석**
  - 기존 시스템: 센서 데이터 단절 시 `ros::Time::now()`로 타임스탬프 대체 발행
  - 데이터 복구 후 센서 타임스탬프가 과거 시간으로 인식되어 TF 트리 시간 역전 발생
- **해결 방법**
  - 센서 타임스탬프 단일화: Odom 발행 시 센서 고유 타임스탬프만 사용하도록 정책 변경
  - LFM 상태 모니터링 활용: 데이터 단절 구간을 상태 모듈에서 인지하여 별도 처리
  - Delta 기반 Prediction: LFM에서 시간 간격(delta) 체크 후 예측 보정 적용
- **결과**
  - `TF_OLD_DATA` 에러 해소 및 TF 트리 안정성 확보
  - LFM 기반 위치추정 파이프라인 신뢰성 입증

> [!WARNING]
> **[문제 상황 2] 부팅 시 초기 위치 미설정으로 인한 포즈 점프 (0, 0, 0 발행)**

- **원인 분석**
  - `robot_localization`의 초기 위치 파라미터 수동 입력 방식의 한계
  - 매 부팅마다 초기 위치를 수동 설정하는 것은 운용 환경에서 비현실적
- **해결 방법**
  - 포즈 영속화 모듈 개발: 로봇의 마지막 위치를 주기적으로 YAML 파일에 저장
  - 자동 초기화 파이프라인 구축: 부팅 시 저장된 YAML을 읽어 초기 위치 파라미터로 자동 주입
  - 기존 런치 시스템과 통합하여 무중단 적용
- **결과**
  - 부팅 시 초기 위치 자동 복원으로 운용 안정성 확보
  - 재시작/재부팅 시나리오에서의 위치추정 연속성 보장

## 4️⃣ 결과 및 성장

**정량적 성과**

- 센서, 컨트롤러, 출력부의 역할을 분리하여 시스템 구조를 모듈화하고, 각 모듈의 상태 점검이 가능하도록 개선
- 센서 데이터 끊김 시 TF 산출이 중단되던 문제를 예측 기반 보정으로 해결하여 안정적인 TF 출력 유지
- TF 출력 주파수를 기존 12Hz에서 설정 가능한 구조로 변경 (기본 30Hz, 약 2.5배 향상)

**정성적 성장**

- 입력 → 처리 → 출력을 단일 프로세스로 구성 시 입력 중단이 전체 시스템 중단으로 이어지는 위험성을 경험하고, 다중 센서 환경에서는 프로세스 분리와 동기화 설계가 필수임을 체득
- 기능별 모듈화를 통해 상황에 맞는 유연한 적용과 개별 모듈 상태 점검이 가능해지며, 디버깅 및 유지보수 효율이 높아짐을 학습
